# Canopy/Native — Master Engineering Plan: A Credible React Native Competitor (Core Engine)

*Lead author synthesis of 7 engineering-track designs · verified against the working tree 2026-06-15 · ecosystem/package-breadth explicitly OUT of scope.*

---

## 1. Goal & Definition of Done

**"A credible React Native competitor (minus ecosystem)" means one concrete thing:** a developer can build a real, performant, multi-screen app — forms, lists, images, modals, navigation between screens — in Canopy's TEA/Elm dialect, run it on **both iOS and Android** with a **Metro-class dev loop** (sub-second edit→see, state preserved), and **ship signed release builds to both stores**, using the core engine plus whatever native capabilities the owner hand-writes. The differentiator is the architecture: not React/Fabric, but Canopy-compiled `view` code → renderer-agnostic `VirtualDom.Node` → direct platform views (`android.view`/UIKit) over a flat-C JSI ABI + Yoga, with the TEA runtime on Hermes.

**The acceptance bar — all must be green:**

| Dimension | Acceptance criterion |
|---|---|
| **Correctness by construction** | The 264-line compiler patch is committed *with tests*; native no longer rides unverified compiler behavior. A bundle never crashes `F7 is not defined`. |
| **Both platforms render parity** | iOS host compiles and passes the full BUILD-AND-VALIDATE Part-5 ledger on simulator + a physical device; iOS capability/component set matches the existing Android set; a shared cross-platform test-vector suite proves the two hand-written hosts do not silently drift. |
| **Dev loop** | Single-module edit → hot reload **<1s p50**, TEA model preserved on same-shape edits, graceful full-reload on Model-shape change, compiler errors as an inline red box, works over USB **and** LAN/remote-box. |
| **Perf at scale** | `lazy` actually memoizes (zero host mutations on a no-op frame); 1000-row list flings within a ratified multiple of RN 0.76.9 on the same device (proposed: ≤1.2× jank, ≤1.3× TTI, ≤1.5× RSS); a per-commit perf regression gate exists. |
| **Release safety** | No release build can load an unsigned `/data/local/tmp` bundle; integrity check fails *closed* in release; R8-minified signed Android release and signed iOS release both boot and exercise every native-reached path; crashes symbolicate to `.can` source. |
| **Ship** | Signed Android `assembleRelease` validated on device; iOS Release archive + TestFlight internal build on a physical device. |
| **RN-version durability** | Hermes/Yoga vendored on *our* cadence behind a checksummed `vendor.lock.json`; a boot-time + CI ABI/bytecode-version gate makes a mismatch a loud, detectable error — never a silent device SIGABRT. |
| **Provision UX** | `provision-and-test <platform> <user@ip>` installs the full toolchain on a bare box and returns a screenshot + log, for both iOS (Mac) and Android (Linux). |
| **CI** | Every commit builds the JS bundle **from source** (via the pinned patched compiler), runs the JS gate (`ci-test.sh` harness set + `canopy test`), builds debug+release APK, runs the Appium smoke flow on an emulator, and gates on the ABI/perf checks. |

**Explicitly NOT in the bar:** matching Expo's hundreds of modules, autolinking, capability fan-out beyond the *existing Android set*. iOS parity means "up to Android's current capabilities," not "chase breadth."

## 2. Strategy in 5 bullets

1. **The compiler is the leverage point.** Every host on every RN version consumes its output. Land the 264-line patch as *tested* features first (it is the reason native boots at all), then make `canopy make --target native` a first-class durable path that owns JS+map+boot+`.hbc` — killing the brittle shell-out + `Bundle.hs` source-map-shift hack. Correctness-by-construction here is what makes the whole bet durable.

2. **Exploit the one structural advantage TEA has over React.** App state is a single `var model` closure in `runtime.js`. Expose a capture/restore + Sub-shutdown seam and a compiler-emitted **Model type-hash**, and a hot reload becomes "just another TEA step": re-seed the model, let the *existing* walker diff path emit the minimal mutation set. This is the entire reason a Metro-class loop is tractable here — but it is hostage to the `lazy` fix (otherwise every reload re-diffs the whole tree).

3. **Two hosts, one ABI, zero silent drift.** The Android host is real and emulator-validated; iOS is ~6k lines of never-compiled ObjC++. Get iOS to first-compile, reach parity gate-by-gate, and lock both hosts to a **shared platform-neutral test-vector suite** plus a **single additive ABI seam** (`__fabric_callMethod`/command) for the imperative surface that is absent on *both* today. Additive ABI changes mean iOS can adopt later without a version bump.

4. **Measure before you optimize; gate everything you measure.** There is *zero* timing data today — only mutation counts in a mock. Build the JS-CPU bench harness, then on-device frame instrumentation, then a head-to-head RN app, then ratify numeric perf gates. Only re-architect the hot path (JSON marshalling, UI-Looper coupling) *after* measurement justifies it. Batch before binary; single-thread+batch before breaking the JS-thread==UI-thread invariant.

5. **Make RN's treadmill irrelevant.** Canopy uses only Hermes + Yoga + JSI — not Fabric. So pin Hermes+Yoga on *our* cadence (≈2×/year), bind Hermes through the **stable C-vtable ABI** (`hermes_abi.h`, already vendored and unused), and make every bump a scripted `revendor.sh` + checksum + boot/CI ABI gate. RN becomes "a convenient bundle of matched binaries," not a release train we ride.

**Three load-bearing cross-track dependencies the schedule lives or dies on:**

1. **DEV-9 (sub-second incremental compile)** — the dev loop's whole value proposition. The risk is entirely in `../compiler`. *Negotiate a warm/persistent compile-server interface with the compiler workstream in week 1*; fall back to whole-program compile + warm process if incremental IIFE linking isn't ready.
2. **The Model type-hash (DEV-8)** — a *new* compiler emission that gates true state-preserving Fast Refresh. If it slips, ship DEV-4's full-reload-preserve-position as the default.
3. **RND-1 lazy fix** — already verified broken. Until it lands, the dev loop demos fine on a counter and degrades on any real screen, and list virtualization is fiction. It's a 0.6-week pure-JS fix; do it first.

## 6. Remote test infrastructure — "give me an IP, it installs everything and tests"

The end-to-end UX is a single command per platform, both keyed on one SSH target, both pulling a screenshot + log back to the Linux dev box. The bundle is **always built locally** (consistent with the existing iOS harness's `CANOPY_BUNDLE` pattern) so the remote box only needs platform toolchain + the bundle.

**`scripts/provision-and-test.sh <platform> <user@ip> [--release] [--device]` [RB-4]:**
1. Writes the appropriate `.remote-build.env` from the IP (user never hand-edits it; unifies the `MAC_SSH`/`LINUX_SSH` naming split).
2. Runs `canopy-native build <app>` **locally** (using the pinned patched compiler from CI-2) → bundle + map + manifest, exports `CANOPY_BUNDLE`.
3. Dispatches to the platform harness `provision && all`.
4. Prints the pulled `remote-artifacts/screen.png` and tails `canopy.log`.

**iOS path — `host/ios/remote-build.sh` (exists; add `provision` [RB-1]):**
`provision` (non-interactive Homebrew + `brew install xcodegen cocoapods node`, verify `xcode-select`; Xcode itself can't be scripted — prints remediation) → `bootstrap` (`npm i react-native@0.76.9` → Hermes/Yoga pods, the load-bearing ABI source) → `gen` (xcodegen + pod install) → `build` (xcodebuild, errors teed to `build.log`) → `run` (simulator, pull `screen.png` + os_log) → `release`/`test` for archive + XCUITest. **Hard-gated on a reachable Mac + (for device/Release/TestFlight) a paid Apple Developer account.**

**Android path — new `host/android/remote-build.sh` [RB-2] (mirror of iOS):**
`provision` (apt `temurin-17` + `sdkmanager` platform-34, build-tools 34.0.0, `ndk;26.3.11579264`, `cmake;3.22.1`, emulator + system-image, write `local.properties`) → `sync` (rsync excluding build/.gradle/.cxx/node_modules) → `bundle` (stage local bundle into `assets/`) → `build` (`assembleDebug`/`assembleRelease`) → `run` (boot AVD `-no-window -gpu swiftshader_indirect` *or* `adb devices` for a connected device; `adb install -r`; launch; `adb exec-out screencap -p` → `screen.png`; filtered `logcat` → `canopy.log`).

**Constraints to surface to the owner:**
- Cheap cloud Linux VMs often lack `/dev/kvm` → emulator falls back to slow software GPU; **real arm64 perf needs a physical device or an arm cloud instance** (build.gradle's `abiFilters` are arm64-v8a + x86_64, but the emulator runs x86_64). Hence **DF-1** (device farm) for credible perf numbers.
- **RB-3 release-safety gate:** `run` clears `/data/local/tmp/canopy.bundle.js` before a release install, and a CI security test asserts a release build ignores it — this test is *red until AND-1's `BuildConfig.DEBUG` guard lands*, so sequence them together.

**CI ties it together [CI-3/CI-4/CI-7/E2E-1]:** every commit builds the bundle from source via the pinned compiler, runs the unified JS gate (`ci-test.sh` + `canopy test`), builds debug+release APK, runs the Appium smoke flow on a KVM emulator, and gates on the ABI [RNV-8] + perf [RND-11] checks.

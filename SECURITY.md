# Security Policy — canopy/native

A "most reliable" framework that vendors a months-old Hermes with no CVE pipeline is a contradiction.
This document is the supply-chain + vulnerability-response contract, and it is partly CI-enforced
(see "Automated gates").

## Supported versions

canopy/native is pre-1.0 and ships from `main`. Security fixes land on `main` and the latest tagged
release. Older tags are not back-patched until a 1.0 support policy exists.

## Reporting a vulnerability

**Do not open a public issue for a vulnerability.** Report privately via GitHub Security Advisories
(repository → Security → "Report a vulnerability") or email the maintainer. Include repro steps and
the affected component (Canopy code path, the JSI/`__fabric_*` ABI, a vendored binary, or a capability
module).

**Response SLA** (best-effort for a small project — stated honestly, see "Bus-factor"):
- **Acknowledge** within 72 hours.
- **Triage + severity** (CVSS-style: critical / high / medium / low) within 7 days.
- **Fix or documented mitigation**: critical ≤ 14 days, high ≤ 30 days, medium ≤ 90 days, low best-effort.
- Coordinated disclosure: a fix ships before public details; reporters are credited unless they prefer otherwise.

## Dependency / supply-chain posture

canopy/native binds a deliberately tiny, **pinned** native surface — Hermes + JSI + Yoga + fbjni from
**React Native 0.76.9**, and **onnxruntime 1.26.0** — recorded with sha256 + provenance in
[`host/vendor.lock.json`](host/vendor.lock.json) and regenerated by `scripts/revendor.sh`. Because
those are vendored binaries (not resolved by a package manager), their CVE tracking is explicit:

1. **SBOM.** `scripts/gen-sbom.sh` emits a CycloneDX SBOM (`sbom.cdx.json`) from `host/vendor.lock.json`
   + the npm/Haskell lockfiles. It is published as a CI artifact on every run.
2. **CVE scan.** The CI `security-scan` job runs `osv-scanner` over the npm/Haskell lockfiles **and**
   the generated SBOM (so a CVE against the vendored Hermes/Yoga/onnxruntime surfaces, not just JS
   deps). Findings are triaged against `osv-scanner.toml` (documented, time-boxed allowlist entries
   only — never a blanket ignore).
3. **Upstream watch.** The scheduled `bump-check` job already pings for newer Hermes/Yoga/onnxruntime;
   a security advisory against a pinned version is a **fix-or-mitigate** trigger under the SLA above,
   not a "wait for the next bump" item.
4. **Dependabot** (`.github/dependabot.yml`) keeps the npm + GitHub-Actions surfaces current.

## Signing-key management

- **Android release keystore** and **iOS distribution cert / provisioning profile** are **never** in
  the repo. They are injected at build time from CI secrets (`CANOPY_KEYSTORE_BASE64`, `APPLE_*`); the
  committed dev keystore is validation-only and fail-closed in release
  (`scripts/check-release-bundle-security.sh`).
- **OTA bundle signing key** (when OTA lands, DXL-3/4): ed25519, generated + held offline; the host
  pins a **set** of public keys so a key can be rotated without bricking installed apps; a leaked key
  is handled by rotating the set + shipping a store update that drops the old key.
- **Rotation / compromise:** on suspected compromise, rotate the affected key, invalidate the old one,
  and publish an advisory. Upload-key compromise on Play is recovered via Google Play App Signing key
  reset; an Apple cert is revoked + re-issued.

## Automated gates (what's enforced, not just promised)

- `security-scan` (CI): SBOM generation + `osv-scanner` over lockfiles + SBOM.
- `vendor-verify` (CI): every vendored binary matches its `host/vendor.lock.json` sha256 (a tampered
  or drifted binary fails the build).
- `abi-gate` (CI): the linked Hermes bytecode version matches the pin (no silent ABI corruption).
- `scripts/check-release-bundle-security.sh` (CI): a release build cannot load an unsigned bundle.

## Bus-factor (honest statement)

canopy/native is currently maintained by one person. There is no 24/7 on-call. The SLA above is a
good-faith target, not a staffed guarantee; the reproducible-build-from-pin (REPRO-1) + the documented
secret/SHA succession plan (GOVERNANCE.md) are the mitigations that let another maintainer take over.
This will be revisited when the project has more than one maintainer or a funding model.
